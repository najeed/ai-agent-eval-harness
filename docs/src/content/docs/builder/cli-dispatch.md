# Command Orchestration

AgentV uses a **Dynamic Routing Registry** to manage its expansive CLI interface. This architecture separates the **CLI Parsing** (handled by `argparse`) from the **Handler Execution**. This decoupling allows for modular extensions, plugin-injected commands, and exhaustive automated testing.

## The `RoutingRegistry`

The `eval_runner.cli.RoutingRegistry` is the central clearinghouse for all harness commands. Instead of a monolithic `if/elif` block, the harness uses a registry pattern to map commands to their implementation functions.

### Registration Pattern

Commands are registered during the `eval_runner.cli` initialization:

```python
from eval_runner.cli import RoutingRegistry

# Core command registration
RoutingRegistry.register("evaluate", handle_evaluate)
RoutingRegistry.register("console", handle_console)
RoutingRegistry.register("certify", handle_certify)
```

### Dynamic Dispatch

When you run `agentv <command>`, the harness:
1.  **Parses** global arguments (e.g., `--debug`).
2.  **Identifies** the target command from the registry.
3.  **Dispatches** the execution to the registered handler function, passing the parsed arguments.

## 🔒 Exhaustive CLI Testing

To ensure industrial-grade reliability, AgentV enforces an **Exhaustive Testing Standard** for the CLI layer. Every registered command must have a corresponding unit test that verifies:
1.  **Successful Dispatch**: The command reaches the handler with the correct parameters.
2.  **Error Handling**: Invalid arguments or environment failures are caught and reported with meaningful exit codes.
3.  **Namespace Integrity**: Plugin commands (registered via `on_register_commands`) do not collide with core engine commands.

### Example: Testing a Dispatcher
We use `unittest.mock` to verify that the CLI correctly routes to the `eval_runner` engine.

```python
def test_cli_routing_evaluate(self):
    with patch("eval_runner.cli.handle_evaluate") as mock_handler:
        # Simulate 'agentv evaluate --path my_path'
        main(["evaluate", "--path", "my_path"])
        mock_handler.assert_called_once()
```

---

## 🔌 Plugin Commandments

Plugins can inject their own commands into the `agentv` namespace while maintaining security.

1. **Scoping**: Plugin commands are automatically prefixed to prevent hijacking core functions.
2. **Lifecycle**: Documentation for plugin commands is dynamically generated by the `RoutingRegistry`, ensuring that `agentv --help` is always accurate, regardless of which plugins are active.

> [!TIP]
> **Staff Engineer Note**: This dynamic dispatch model is what allows AgentV to scale to hundreds of sub-commands across 30+ industrial sectors without becoming a "spaghetti" maintenance nightmare.
