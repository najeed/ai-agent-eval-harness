The **Trust Protocol** establishes a non-repudiable audit trail for AI agent evaluations. By using asymmetric cryptography (**ED25519**), traces generated by the harness are "sealed" to ensure they haven't been tampered with and genuinely originate from a trusted environment.

---

## 🏗️ The 5-Phase Lifecycle

For the **Security Auditor**, the protocol ensures end-to-end integrity through five distinct phases:

### 1. Evaluation
The agent is tested against a scenario. During execution, the **Environmental DNA** (Registry Snapshot) is captured and hashed.

### 2. Certification (`certify`)
The final `run.jsonl` trace is sealed. The `certify` command generates a **Verification Certificate (VC)** containing:
- SHA-256 hash of the trace file.
- Environmental DNA hash.
- ED25519 asymmetric signature.

### 3. Verification (`verify`)
Stakeholders use the **Public Key** to verify the VC. This proves the trace exactly matches the one generated in the secure environment.

### 4. Gating (`gate`)
The **Hard Gate** step in CI/CD. The process exits with a non-zero code if:
- The signature is invalid.
- The Git commit hash doesn't match the metadata.
- The agent failed critical safety or performance metrics.

### 5. Drift Detection (`import-drift`)
Production traces are compared against the certified baseline to identify behavioral divergence (Drift) before they become security vulnerabilities.

---

## 🔑 Key Management

The harness supports **Enterprise-Grade Key Management**:
- **Private Key**: Used by the Runner (e.g., in a secure CI/CD Runner) to sign traces.
- **Public Key**: Shared with auditors or third parties for verification.

> [!TIP]
> MultiAgentEval supports **Enterprise KMS/HSM** integrations. Private keys can be loaded directly from secure hardware modules rather than local `.pem` files.

---

## 🚧 CI/CD Integration

The `gate` command is the industrial standard for enforcement.

```bash
multiagent-eval gate \
  --run-id 2026-04-09_tx_99 \
  --public-key keys/prod_public.pem \
  --hash ${{ github.sha }}
```

### Verification Failure Modes
If the gate fails, the auditor is provided with a forensic signature mismatch analysis:
- **`SIGNATURE_MISMATCH`**: The trace was modified post-run.
- **`ENVIRONMENT_DIVERGENCE`**: The world shims used at test-time don't match the sanctioned baseline.
- **`PII_EXFILTRATION`**: Security filters triggered a post-run redaction failure.
